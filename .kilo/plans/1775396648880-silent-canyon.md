# Refactor plan: decouple game engine from persisted game data

## Goal
Split runtime behavior (engine) from serializable state (DTO/data model) so Durable Object storage, websocket events, and game logic are no longer tightly coupled to mutable class instances.

## Current coupling observed
- `src/engine/game.ts` mixes:
  - domain state (`id`, `topic`, questions, players, score, state)
  - behavior (`nextQuestion`, `answer`, scoring, transitions)
  - infra-ish concerns (listener wiring via `Player`, scheduler dependency)
- `src/question/question.ts` mixes:
  - persisted fields (text, answers, timestamps, player answers)
  - behavior (`start`, `answer`, `getScore`, parsing/cloning)
- `src/game-do.ts` persists whole class graphs with `JSON.stringify(game)` and restores via `Game.fromState(...)`, which forces state format to mirror engine internals.

## Naming direction (aligned)
- Keep data/state names simple: `Game`, `Question`, `Player`, `PlayerAnswer`.
- Name behavior classes explicitly as engines: `GameEngine`, `QuestionEngine`.
- Introduce a runtime connection wrapper for websocket/listener concerns: `PlayerSessionHandler`.
- Avoid `*Dto` suffix in runtime code paths unless needed for external API contracts.

## Target architecture

### 1) Pure data layer (serializable only)
Create dedicated serializable data types/classes with no runtime dependencies:
- `Game`
- `Question`
- `Player`
- `PlayerAnswer`

Recommended location:
- `src/model/game.ts`
- `src/model/question.ts`

Rules:
- Data objects use primitives/arrays/records only.
- Use `string` for date/timestamps in storage objects (or epoch number), not `Date` objects.
- No websocket listeners, scheduler, or runtime-only references in persisted data objects.
- Reuse existing classes when they are already data-only; otherwise split runtime fields out.

### 2) Domain state + engine split
Create a runtime state holder and an engine service around it:
- `Game` as mutable runtime state model
- `GameEngine` (methods: `addPlayer`, `setSpectator`, `nextQuestion`, `answer`, `timeUp`, scoring, event emission)
- `QuestionEngine` for question-specific behavior (`start`, `answer`, `distance`, `score`)
- `PlayerSessionHandler` (runtime-only mapping from `playerId` to `GameEventListener` and session metadata)

Recommended location:
- `src/engine/game-engine.ts`
- `src/engine/question-engine.ts`

`GameEngine` constructor should accept:
- `state: Game`
- `timeScheduler: TimeScheduler`
- `playerSessions: PlayerSessionHandler`

### 3) Mapper layer between persisted shape and runtime model
Create explicit mapping utilities:
- `gameFromData(data, timeScheduler, listenersByPlayerId?) -> GameEngine`
- `gameToData(engine/state) -> Game`
- `questionFromData/questionToData`

Recommended location:
- `src/model/mapper/game-mapper.ts`
- `src/model/mapper/question-mapper.ts`

This isolates persistence format from engine internals.

## Implementation steps

1. **Introduce data models and mappers first (no behavior changes yet)**
   - Add serializable data definitions for game/question/player answer/player.
   - Add mapper functions with full field coverage.
   - Keep existing `Game`/`Question` logic operational while wiring conversion helpers.

2. **Extract Question behavior into engine-focused class**
   - Keep `Question` as data model; move behavior to `QuestionEngine`.
   - Remove `fromQuestionObj`/`fromState` style persistence constructors from behavior class.
   - Ensure time handling is explicit and consistent (avoid mixed `Date`/string ambiguity).

3. **Extract Game behavior into `GameEngine` over `GameStateModel`**
   - Move transition/scoring/answer flow from current `Game` into engine.
   - Move `eventListenersByPlayerId` into `PlayerSessionHandler` runtime component.
   - Keep `timeScheduler` runtime-only (not persisted).

4. **Refactor Durable Object persistence flow**
   - In `src/game-do.ts`, replace direct `JSON.stringify(game)` with:
     - `const data = gameToData(engine)`
     - `storage.put("game", JSON.stringify(data))`
   - On load:
     - parse persisted data
     - instantiate `PlayerSessionHandler`
     - instantiate engine via mapper + scheduler + `PlayerSessionHandler`
   - Rehydrate websocket listeners by session id exactly as done today, but via `PlayerSessionHandler` only.

5. **Update dependent modules**
   - `src/event/game-message-handler.ts` should depend on `GameEngine` interface/class.
   - `src/event/game-event.ts` can continue exposing question data, but preferably emit DTO-friendly payloads (or read-only view models) instead of full engine objects.
   - `src/question/question-generator.ts` should create question DTOs (or model objects + mapper), not engine instances directly.

6. **Compatibility + validation**
   - Add a one-time backward-compat loader in DO init:
     - try parse as new data shape
     - fallback parse old stored shape and map to new model
   - This prevents breaking existing persisted games.

7. **Hardening and tests**
   - Add focused tests for:
     - data roundtrip (`engine -> data -> engine`)
     - scoring consistency before/after refactor
     - state transitions (`PREPARE -> QUESTION -> BETWEEN_QUESTIONS/FINISH`)
     - alarm + answer race behavior
   - Verify websocket join/reconnect path still rebinds listeners correctly.

## Suggested concrete file migration
- Keep existing files during transition:
  - `src/engine/game.ts` -> gradually replaced by `src/engine/game-engine.ts`
  - `src/question/question.ts` -> split into data + engine components
- Add new files:
  - `src/model/game.ts`
  - `src/model/question.ts`
  - `src/model/mapper/game-mapper.ts`
  - `src/model/mapper/question-mapper.ts`
  - `src/engine/question-engine.ts`
  - `src/engine/player-session-handler.ts`

## Key design decisions to enforce during implementation
- Persistence schema must not depend on class constructors.
- Engine classes must be instantiable purely from data objects + runtime dependencies.
- Runtime dependencies (`WebSocketGameEventListener`, `TimeScheduler`) are injected and excluded from storage.
- Event payloads should not leak mutable engine internals.
- `Player` is pure data only; runtime websocket/listener references live exclusively in `PlayerSessionHandler`.

## Risks and mitigations
- **Risk:** breaking live DO data format.
  - **Mitigation:** dual loader (new DTO first, legacy fallback) for at least one migration window.
- **Risk:** hidden coupling in event payloads.
  - **Mitigation:** introduce explicit event view type and map from runtime state.
- **Risk:** behavior drift after extraction.
  - **Mitigation:** parity tests around scoring and transitions before deleting old classes.

## Expected outcome
After this refactor, game logic remains in engine classes (`GameEngine`, `QuestionEngine`), persisted game data is represented by stable data objects (`Game`, `Question`, `Player`, `PlayerAnswer`), and Durable Object restore/save becomes deterministic and independent from class implementation details.

## Test implementation plan (2-player full game)

### Test stack
- Use `jest` + `ts-jest` for TypeScript unit/integration-style engine tests.
- Add scripts:
  - `test`: `jest --runInBand`
  - `test:watch`: `jest --watch`
- Keep tests isolated from Cloudflare runtime by testing `GameEngine` directly with fakes.

### Files to add
- `jest.config.cjs`
- `src/test/fakes/fake-time-scheduler.ts`
- `src/test/fakes/recording-game-event-listener.ts`
- `src/engine/game-engine.spec.ts`

### Test doubles
- `FakeTimeScheduler`
  - captures scheduled `questionId` and `delay`
  - satisfies `TimeScheduler` interface
  - no timers/alarms, synchronous and deterministic
- `RecordingGameEventListener`
  - captures all emitted `GameEvent` values into array
  - exposes helper methods for assertions (last event, event count)

### Main end-to-end game-flow test
Create one high-value scenario: **"plays a complete 2-player game and finishes with expected state and score"**

Given:
- game with 2 questions:
  - Q1 numeric (no alternatives)
  - Q2 multiple-choice with alternatives including correct answer
- 2 players join (`p1`, `p2`) with recording listeners

When:
1. `nextQuestion()` for Q1
2. both players answer Q1 (different distances/timing to verify ranking)
3. verify question auto-finish when all non-spectator players answered
4. `nextQuestion()` for Q2
5. both players answer Q2 (one correct, one incorrect)
6. verify transition to `FINISH` after second question

Then assert:
- state transitions include `QUESTION`, `BETWEEN_QUESTIONS`, final `FINISH`
- `unusedQuestions` is empty and `currentQuestion` is `null`
- `usedQuestions.length === 2`
- scheduler was called for each asked question with right ids/delays
- final score map has both players and expected values
- both listeners received game events through the session handler path

### Additional focused tests
- **Spectator exclusion test**
  - mark one player spectator
  - only non-spectator required for `allPlayerAnswered`
  - spectator not present in score map
- **timeUp behavior test**
  - call `timeUp` with current question id
  - question transitions to `BETWEEN_QUESTIONS` or `FINISH`
- **wrong-question-id answer rejection**
  - answer using stale/invalid question id
  - returns `false`, no score/state mutation
- **persisted session mapping continuity**
  - serialize sessions via `PlayerSessionHandler.serializeSessions()`
  - rebuild engine from stored state
  - ensure session ids are reattached as dummy listeners

### Assertion style guidance
- Assert observable outcomes over internals:
  - public engine methods
  - resulting game data state
  - emitted events
- Avoid brittle assertions on exact full event payload snapshots unless needed.

### Execution and success criteria
- Run `npm test` and ensure all tests pass.
- Keep test runtime fast (<2s local) and deterministic (no real clock waits).
- Ensure tests do not import Durable Object classes or Cloudflare globals.
